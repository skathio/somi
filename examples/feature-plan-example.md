# Plan — Per-team webhook rate limiting

> Owner: somi-ai (illustrative) • Status: `draft`
> Created: 2026-05-20

## 1. Problem statement

We currently expose `POST /webhooks/ingest` as a single global endpoint with one shared rate limit
(1000 req/min). A noisy customer can starve quieter customers. We need per-team rate limiting, audit
logging of denied requests, and an operator-controlled kill switch to halt ingestion for a specific
team in an incident.

## 2. Goals / Non-goals

**Goals**
- Per-team request budget (configurable, default 200 req/min).
- Denied requests return `429` with `Retry-After` and a structured error body.
- Audit log entry on every deny: team, time, count-in-window.
- Kill switch (ops-only) that disables ingestion for a specific team without a deploy.

**Non-goals**
- Per-endpoint sub-limits within the team budget. (Future work; out of scope here.)
- Cost-based limiting (request size, payload complexity). Time-based only for now.
- Dashboards beyond the basic metric we expose; that's a separate ticket.

## 3. Assumptions

- [ ] We have a Redis cluster available with the same SLA as the webhook service.
- [ ] Teams are already identifiable from a header (`X-Team-Id`) authenticated upstream.
- [ ] The kill switch can live in our existing config store; no new infra needed. **Needs confirmation.**

## 4. Unknowns / Decisions needed

- **Rate-limit algorithm**: fixed window vs. token bucket. Recommend **token bucket** for smoother
  bursts. Decision: <pending — ask SRE>.
- **Default limit**: 200/min suggested based on a quick log query of p95 traffic; needs confirmation
  from product. **Decision needed before phase 2.**

## 5. Architecture sketch

```
[Client]
  -> [WebhookHandler]
        -> [RateLimiter (Redis-backed token bucket)] -- denied --> 429 + AuditLog
        -> [KillSwitch (config-store boolean)]       -- on      --> 503 + AuditLog
        -> [IngestionQueue]
              -> [Worker]
```

The limiter and the kill-switch are independent gates; both must pass.

## 6. Tradeoffs considered

| Option | Pros | Cons | Verdict |
|--------|------|------|---------|
| Token bucket in Redis | Smooth bursts, well-understood, ops-friendly | Network hop per request | **Chosen** — fits our latency budget |
| In-process leaky bucket | Zero extra deps | Drift across replicas, unfair under autoscaling | Rejected |
| Sliding-window in Redis | More precise | More expensive script per request | Rejected for now; revisit if bucket drift causes complaints |

## 7. Phases

### Phase 1 — Limiter contract + in-memory impl (for testing)
- **Goal**: define interface, prove behavior, no integration with webhook yet.
- **Exit criteria**: `RateLimiter` interface defined, in-memory bucket passes table-driven tests.

### Phase 2 — Redis-backed limiter + integration with WebhookHandler
- **Goal**: real limiter wired into the handler behind a flag.
- **Exit criteria**: webhook returns 429 for over-limit traffic in staging; flag-gated rollout.

### Phase 3 — Kill switch + audit log
- **Goal**: operator-controlled kill switch + structured audit log entries on every deny.
- **Exit criteria**: ops runbook updated; verified in staging.

## 8. Iteration slices (per phase)

### Phase 1 / Iteration 1 — `RateLimiter` interface + in-memory impl
- **Scope**: define interface; implement `InMemoryLimiter`; table tests.
- **Files**: `internal/ratelimit/limiter.go` (new), `internal/ratelimit/inmem.go` (new),
  `internal/ratelimit/inmem_test.go` (new).
- **Acceptance**: interface `Allow(ctx, key, n) (bool, retryAfter, error)`; tests cover burst,
  expiry, concurrency, zero-budget.
- **Tests**: table-driven unit tests with a frozen clock.
- **Observability**: none yet (no integration).
- **Rollback**: revert the PR.

### Phase 2 / Iteration 1 — `RedisLimiter` implementation
- **Scope**: Redis-backed token bucket using Lua script (atomic refill + take).
- **Files**: `internal/ratelimit/redis.go` (new), `internal/ratelimit/redis_test.go` (new — integration).
- **Acceptance**: integration test against a real Redis (testcontainers or docker-compose) passes;
  concurrent callers respect the budget.
- **Tests**: integration test for atomicity under concurrent calls.
- **Observability**: log on Redis errors (degraded mode falls open by default for now — see §10).
- **Rollback**: revert the PR; in-memory limiter still works for tests.

### Phase 2 / Iteration 2 — Wire limiter into WebhookHandler behind a flag
- **Scope**: inject `RateLimiter` into `WebhookHandler`; reject with 429 + `Retry-After`; flag-gated.
- **Files**: `internal/webhook/handler.go` (modified), `internal/webhook/handler_test.go` (modified).
- **Acceptance**: with flag on, over-limit team gets 429; with flag off, behavior unchanged.
- **Tests**: handler tests with both flag states; e2e smoke test in staging.
- **Observability**: `webhook.ratelimit.allow` and `webhook.ratelimit.deny` counters labeled by
  team_id_bucket (NOT raw team_id — cardinality discipline).
- **Rollback**: flag off.

### Phase 3 / Iteration 1 — Kill switch
- **Scope**: read kill-switch config per team on the request path; return 503 + `Retry-After: 300`
  when set.
- **Files**: `internal/webhook/handler.go` (modified), `internal/config/killswitch.go` (new).
- **Acceptance**: flipping the config rejects new requests within 30s without a deploy.
- **Tests**: handler test for the killed path.
- **Observability**: `webhook.killswitch.tripped` counter labeled by team_id_bucket.
- **Rollback**: clear the config; behavior reverts to limiter-only.

### Phase 3 / Iteration 2 — Structured audit log on deny / kill
- **Scope**: emit a structured log on every deny / kill with team_id (real), correlation_id, reason,
  window_used.
- **Files**: `internal/webhook/audit.go` (new), `internal/webhook/handler.go` (modified).
- **Acceptance**: log lines visible in the central log sink; ops runbook updated.
- **Tests**: handler test asserting log line shape.
- **Observability**: itself is observability.
- **Rollback**: revert the PR.

## 9. Test strategy

- **Risk-driven**: focus on (a) atomicity under concurrent requests (real Redis), (b) clock-driven
  expiry edge cases (frozen clock), (c) handler returning correct status + headers, (d) audit log
  shape.
- **Unit**: limiter algorithm, handler decision logic.
- **Integration**: Redis limiter with a real Redis container.
- **E2E**: staging smoke test with flag on.
- **Mocks**: none of Redis (it's a real container in integration). Clock is injected; tests use a
  fake clock.
- **Determinism**: frozen clock for unit; container snapshot for integration.

## 10. Security considerations

- **Trust boundary**: untrusted webhook payload from the internet.
- **Sinks engaged**: log line (log-injection risk if we put raw team-supplied headers in logs), Redis
  Lua script (no concat — args parameterized).
- **OWASP**: A04 (insecure design — abuse via DoS), A05 (misconfiguration if killswitch wide-opens by
  default), A09 (logging — must capture without leaking PII).
- **Mitigation in plan**:
  - Limiter applies before body parsing (cheap-reject; avoids resource exhaustion).
  - Killswitch defaults to **off** (limiter still applies); never wide-open by default.
  - Audit log doesn't include request body — just team_id, correlation_id, reason, counts.
  - `security-reviewer` runs at end of Phase 2 (before flag is turned on in prod).

## 11. Observability plan

- **Logs**: structured audit log on every deny/kill with fields `team_id`, `correlation_id`,
  `reason`, `window_used`.
- **Metrics**: `webhook.ratelimit.allow{team_bucket}`, `webhook.ratelimit.deny{team_bucket}`,
  `webhook.killswitch.tripped{team_bucket}`. **Note**: team_bucket is a coarse hash of team_id to keep
  cardinality bounded.
- **Traces**: span attribute `ratelimit.outcome=allow|deny|killed` on the existing webhook span.
- **Alerts**: page on deny-rate > 50% sustained for >5min on any single team_bucket (probably an
  attack or misconfig).
- **Dashboards**: existing webhook dashboard adds two panels: per-team-bucket allow/deny rate.

## 12. Rollout & rollback

- **Flag**: `webhook_ratelimit_v1` per environment, default off. Ramp: 1% → 10% → 50% → 100% with
  24h between each.
- **Deploy**: standard rolling deploy; no migration required.
- **Migrations**: none.
- **Rollback**: flag off.
- **Success metric**: zero P0 incidents attributable to the limiter for 14 days post-ramp;
  per-team latency p95 unchanged within ±5ms.

## 13. Risks

| Risk | Likelihood | Impact | Mitigation | Owner |
|------|-----------|--------|------------|-------|
| Redis hiccup → false denies | M | H | Fail-open with circuit breaker; log spike of Redis errors. Threshold + alert. | <eng> |
| Limit too tight for legitimate burst customers | M | M | 24h ramp + per-team override config | <eng> |
| Killswitch misconfig kills all teams | L | H | Killswitch is per-team; no "all" key; ops runbook + 2-person approval | <ops> |
| Clock skew across replicas | L | L | Token bucket is Redis-side; clock authority is Redis | <eng> |
